# Accessibility and Responsive Readiness Audit

**Date:** July 24, 2026
**Purpose:** A direct, evidence-based audit of mobile/tablet layout, keyboard and focus-order behavior, screen-reader labeling, color-contrast/Daylight Lantern light-mode risk, state-handling gaps, and navigation dead-ends across every signed-in beta-critical route. This is read-only inspection. No app code, CSS, database, Vercel, Supabase, or production settings were changed to produce it.

**Method:** direct inspection of `/Users/christine.smith/krovos` (read only): `app/layout.tsx`, and the page files for every route in scope (`app/life-compass`, `app/life-profile`, `app/life-graph`, `app/life-goals`, `app/retirement`, `app/net-worth`, `app/life-labor`, `app/life-calendar`, `app/krovos-guide`, `app/settings`, `app/tools`, `app/life-phase-guides`, `app/life-phase-guides/[slug]`), cross-referenced against `product/beta-ui-acceptance-standards-2026-07-24.md`, `product/beta-readiness-test-plan-2026-07-24.md`, `product/krovos-guide-beta-readiness-audit-2026-07-24.md`, and `product/life-calendar-beta-readiness-audit-2026-07-24.md`. Every finding is labeled **confirmed** (traced to specific code) or **needs live browser testing** (this audit's static reading cannot settle the question, most often because color handling, focus order, and rendered contrast depend on runtime behavior, imported child components not read line by line in this pass, or actual visual rendering no grep can substitute for).

## 1. Cross-Cutting Findings (Apply Across Multiple Routes)

### 1a. The universal navigation shell is real and confirmed
`app/layout.tsx` renders `<GlobalNav />` once, inside a `ThemeProvider`, with `data-theme="dark"` set at the root. No individual route file in this audit imports `GlobalNav` itself, confirming navigation is genuinely shell-owned, not per-page, matching the July 23, 2026 universal-app-shell build already documented in the canonical record. This is a strong, confirmed positive: no route in scope can produce a missing-nav or duplicate-nav dead end by omission, since none of them render their own.

### 1b. A specific, high-confidence Daylight Lantern light-mode risk, found in the highest-traffic routes
**Confirmed:** `app/life-compass/page.tsx`, `app/net-worth/page.tsx`, `app/life-goals/page.tsx`, and `app/life-graph/page.tsx` all use React inline `style={{...}}` objects extensively (136, 13, 56, and 41 occurrences respectively) containing literal hardcoded hex color values, for example `app/life-compass/page.tsx` line 290: `style={{ fontFamily: 'var(--font-fraunces)', fontSize: '1rem', fontWeight: 600, color: '#E9EEF2', marginBottom: '0.2rem' }}`. **None of these four files import or reference `useTheme`, `ThemeToggle`, or any theme-conditional logic anywhere in the file** (confirmed by a direct search for those identifiers, zero matches in all four). `app/retirement/page.tsx` shows the same pattern: zero theme-conditional logic found at the page-file level.

**Why this matters specifically:** an inline `style` attribute has higher CSS specificity than any class-based stylesheet rule, including whatever global mechanism implements the Daylight Lantern light-mode swap for `bg-[#1B2230]`-style Tailwind arbitrary-value classes elsewhere in the app. A global stylesheet override keyed on `[data-theme="light"]` cannot reach into a React inline style object without either `!important` (fragile and not typically how Tailwind-based theming works) or per-component conditional logic choosing a different literal value at render time, neither of which this audit found present in these five files.

**What this audit cannot confirm from a page-level read alone:** several of these pages import child components (for example, Life Compass imports `UpcomingMilestones`, `ProgressAndWins`, `DebtLedger`, `ChildcareBubble`, `ActiveModesDisplay`; Retirement's visual content may live substantially in imported chart/workbench components) that were not read line by line in this pass. It is possible some or all of the actual color-bearing markup is delegated to a child component that does handle theming correctly, which would narrow this finding's real-world impact considerably. **This is exactly the kind of question that requires live browser testing, not further static reading, to resolve with confidence:** load each of these five routes with the theme toggle set to Daylight Lantern and check specifically for light-on-light or otherwise illegible text, especially around the literal `color: '#E9EEF2'`-style values this audit found directly in the page files themselves (which, unlike the child-component question, are not ambiguous, they are confirmed present in the page's own render output).

### 1c. No responsive breakpoint classes found in several data-dense routes
**Confirmed, via a direct search for `md:`, `sm:`, and `lg:` Tailwind prefixes:** `app/life-compass/page.tsx`, `app/life-graph/page.tsx`, `app/life-goals/page.tsx`, `app/net-worth/page.tsx`, and `app/life-labor/page.tsx` contain **zero** matches for any of these three responsive prefixes. `app/retirement/page.tsx`, `app/life-calendar/page.tsx`, `app/settings/page.tsx`, `app/tools/page.tsx`, and both `app/life-phase-guides` files do contain some. The absence of breakpoint classes is not automatically a bug, a layout built entirely from flexbox with wrapping and relative units can be mobile-safe without ever needing a breakpoint prefix, but it removes one of the simplest signals this audit has for confirming deliberate responsive design, and several of the zero-match files (Life Compass, Life Graph, Net Worth) also contain fixed pixel-width literals in inline styles (for example, Life Compass has three `width: '...px'`-style literals, Life Graph has five), which is a much stronger, specific risk signal for phone-width overflow than the mere absence of a breakpoint class. **Needs live browser testing at phone width (roughly 375px) for every route named in this paragraph.**

### 1d. Interactive `<div onClick>` patterns are rare and, where found, appropriately handled
**Confirmed:** a search for `<div ... onClick` across the highest-traffic routes found only three instances total: one backdrop-dismiss overlay in `app/life-graph/page.tsx` (line 278, closes an open node panel on outside click, a low-risk, supplementary dismiss pattern, not a primary control), and two in `app/tools/page.tsx` (lines 497-498) which are the locked-tool dialog itself, correctly marked `role="dialog" aria-modal="true" aria-labelledby="locked-tool-title"`, plus its inner click-stop-propagation wrapper, a standard, correctly-implemented modal pattern. **No route in this audit was found using a bare, unlabeled clickable `<div>` as a primary interactive control**, which is the pattern that would most seriously break keyboard operability if present. This is a confirmed-safe finding, not a gap.

### 1e. Explicit empty-state copy exists on some routes but was not found on others via direct string search
**Confirmed present:** Life Compass ("No goals set yet"), Net Worth ("No asset data yet"), and Life Labor ("No facts saved yet") all contain explicit, member-facing empty-state strings. **Not found via the same search pattern:** Life Graph, Life Goals, Retirement, Settings, and Tools. This does not confirm those five routes lack an empty state, different phrasing would not match the search pattern used, but it means this audit cannot confirm one exists either, and each should be checked directly.
