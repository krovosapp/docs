# Data Lifecycle and Account Deletion Audit

**Date:** July 24, 2026
**Purpose:** A direct, evidence-based audit of what actually happens to a member's data at every stage of the account lifecycle, cancellation, grace period, loss of access, and deletion, against what the product's own privacy language promises. This is read-only inspection. No code, database policies, storage objects, secrets, deployment settings, Stripe, or Supabase configuration were changed to produce it.

**Method:** direct inspection of `/Users/christine.smith/krovos` (read only): `app/privacy/page.tsx`, `app/api/stripe/webhook/route.ts`, `app/api/cron/grace-period-check/route.ts`, `app/api/beta/signup/route.ts`, `lib/adult-migration.ts`, every `.storage.from(` call in the repository, and every `supabase/migrations/*.sql` file referencing `auth.users`, `documents`, or a Storage bucket. Every claim below is either **confirmed** (traced to a specific file and line of behavior) or explicitly marked as needing **live verification** where this codebase, per its own canonical current-state record, has six tables (`profiles`, `reports`, `life_graph`, `waitlist`, `documents`, `conversations`) that predate the tracked-migration convention and have no `CREATE TABLE` statement this audit can read.

**Headline finding, stated up front:** there is no automated account-deletion process anywhere in this codebase. A repository-wide search for any account-deletion route, function, or admin action found none, only a rollback path in `beta/signup` that deletes a just-created auth user if profile setup fails moments later, which is not the same thing. The cancellation-to-grace-period pipeline is real and correctly timed to 30 days, but it dead-ends at a status flag with no code anywhere that reads it and acts. Separately, several tables have no cascade-delete relationship to `auth.users` at all, which would block, not silently skip, a deletion attempt if one were performed today. And no code anywhere in the repository ever deletes a Storage object. These findings compound: even a fully manual deletion, performed by hand through the Supabase dashboard, would likely fail outright for any member who has used Life Labor or a household connection, and would leave every uploaded document and feedback screenshot in Storage regardless of whether the database rows referencing them are ever removed.

## 1. The Lifecycle Chain, As It Actually Runs

**Confirmed, from `app/api/stripe/webhook/route.ts` and `app/api/cron/grace-period-check/route.ts`:**

1. **Cancellation requested** (either the in-app `/cancel` flow or a direct Stripe action): the subscription is set to `cancel_at_period_end: true`; `profiles.subscription_status` remains `active` and `subscription_cancel_at_period_end` is set to `true`. The member retains full access through the paid period, matching the product's own stated grace-period policy.
2. **Stripe actually deletes the subscription** at period end, firing `customer.subscription.deleted`. The webhook handler sets `subscription_status: 'cancelled_grace_period'`, `subscription_cancelled_at`, and `grace_period_ends_at` to exactly 30 days from that moment (`graceEndsAt.setDate(graceEndsAt.getDate() + 30)`), and sends the cancellation confirmation email. `cancelled_grace_period` is **not** one of the access-granting statuses `proxy.ts` checks (`active`, `trialing`, `founder`, `connected_user`, unexpired gift), so **app access is lost immediately at this step**, not after the 30-day window. The 30-day window that follows is a data-retention grace period, not a continued-access grace period.
3. **The daily `grace-period-check` cron** finds every profile with `subscription_status = 'cancelled_grace_period'` whose `grace_period_ends_at` has passed, and updates it to `subscription_status = 'eligible_for_deletion'`.
4. **Nothing further happens, ever, anywhere in this codebase.** A repository-wide search for `eligible_for_deletion` found exactly one match: the cron that sets it. No route, cron, or scheduled job reads that status and takes any deletion action. The pipeline is real and correctly timed, but it currently produces a label, not an outcome.

**What this means concretely:** a member who cancels today will lose access on schedule, will be correctly labeled `eligible_for_deletion` 30 days later, and will then remain in that state indefinitely unless a human takes a manual action this audit found no documented, scripted, or code-assisted path for.

## 3. Are Cascade Deletes, Storage Deletion, and Background Cleanup Actually Wired and Safe?

**Cascade deletes: partially wired, and the gaps are severe enough to block deletion entirely for common accounts.** A clear majority of tables added since the migration convention began (`core_tool_results`, `guide_conversations`/`guide_messages`/`guide_usage_monthly`/`guide_plans`, `calendar_connections`/`calendar_events`, `retirement_plans`, `caregiver_ledgers`/`child_future_contributions`, `net_worth_snapshots`, `payday_checkins`, `user_guide_purchases`) correctly cascade from `auth.users`. But the entire Life Labor table cluster (six tables) and `household_connections` do not, and both are widely-used, Core-adjacent, on-by-default features. **This is not a "some data survives" gap; it is a "the deletion operation itself fails" gap**, since Postgres's default foreign-key behavior on an unspecified `ON DELETE` clause is `NO ACTION`, which raises an error and aborts the entire deleting transaction rather than silently proceeding.

**Storage deletion: not wired at all.** Confirmed by an exhaustive search of every `.storage.from(` call in the repository: only `upload()` and `download()` operations exist. No `remove()` call, no cleanup route, no cron. Every document a member has ever uploaded, and every feedback screenshot, remains in Storage indefinitely regardless of subscription status, grace period, or any database-level deletion that might otherwise succeed.

**Background cleanup: does not exist past the labeling step.** The `grace-period-check` cron is the only automated process in this lifecycle, and it stops at setting a status flag. No cron, queue, or scheduled job in this codebase consumes `eligible_for_deletion`.

## 4. Does the 30-Day Retention Promise Match Implementation?

**The timing math is correct and intentional.** `subscription_cancelled_at` plus exactly 30 days produces `grace_period_ends_at`, and the cron correctly waits for that date before flagging eligibility. This part of the implementation matches the privacy page's stated window precisely.

**The promise itself is not automated.** `app/privacy/page.tsx` reads: "You may export your data or request account deletion by contacting us at privacy@krovos.app. Upon account deletion, your personal data and uploaded documents will be permanently removed within 30 days." This is written as, and is consistent with, a manually-fulfilled promise (a member emails a request; a human acts on it), not a claim that the product deletes data automatically. Read that way, there is no direct contradiction in the copy itself. **The mismatch is that this audit found no evidence the manual process, whatever it is, can actually succeed today**: a real deletion attempt would need to first manually clear every Life Labor table and any `household_connections` row for that user (or that user's household partner), manually remove every Storage object the member ever uploaded, and separately resolve the `gift_entitlements` `RESTRICT` constraint for any member who has redeemed a gift, all before an `auth.users` row deletion could complete without erroring. None of that sequence is documented, scripted, or referenced anywhere this audit could find.

## 5. Can Documents, Raw IDs, or Calendar Data Remain After a Member Believes Their Account Is Gone?

**Documents: yes, confirmed, with certainty.** Since no code ever deletes a Storage object, every uploaded document remains in Storage regardless of what happens to the database rows, including in the scenario where a database-level deletion does succeed for a member with no Life Labor or household-connection rows to block it.

**Raw IDs and cross-references: likely, for common account shapes.** Because the Life Labor cluster and `household_connections` block deletion outright rather than silently completing around them, the most probable real-world outcome for a manually-attempted deletion today is not "some data lingers" but "the deletion operation itself fails with an error," leaving the member's entire account, Life Graph, tool results, Guide conversations, and everything else fully intact, exactly as if nothing had been attempted, unless the person performing the deletion manually clears the blocking tables first and happens to know to do so.

**Calendar OAuth tokens and imported events: no, not if the deletion actually succeeds.** These are the one category audited here with clean, confirmed cascade behavior; if an `auth.users` row deletion completes at all, `calendar_connections` (encrypted tokens included) and `calendar_events` are removed with it. The risk here is entirely upstream, whether the deletion can complete in the first place, not this table's own behavior.

**Life Gaps and waitlist records: yes, by design, and this is worth naming explicitly.** These predate and are not linked to a Krovos account via `user_id`. Deleting a Krovos account does nothing to a same-email Life Gaps submission or waitlist entry; a member would need to separately unsubscribe or request removal of that specific record.

## 6. Prioritized Beta-Readiness Checklist

### Active privacy/data-retention gaps (confirmed, require an implementation decision and code)

1. **No account-deletion process exists anywhere in the codebase.** The entire cancellation-to-eligibility pipeline dead-ends at a status label. Building the actual deletion (or anonymization) action is the single highest-priority item in this audit.
2. **No code ever deletes a Storage object.** Uploaded documents and feedback screenshots persist indefinitely today, independent of any other fix.
3. **Six tables (`life_labor_settings`, `life_labor_tasks`, `life_labor_seasons`, `life_labor_snapshots`, `life_labor_facts`, `life_labor_task_occurrences`) and `household_connections` have no cascade-delete relationship to `auth.users`**, meaning a deletion attempt for a large share of real accounts will fail outright rather than complete partially. This needs either an added `ON DELETE CASCADE` (for tables that should be simply removed) or an explicit pre-deletion cleanup step (if manual review of those rows is intentionally required before deletion, e.g., for a household connection where the other party's data must not be affected).
4. **`gift_entitlements.recipient_user_id`'s `ON DELETE RESTRICT` will permanently block deletion for anyone who has ever redeemed a gift**, unless the deletion process is explicitly designed to anonymize personal data while retaining this specific financial record, a decision this audit found no evidence has been made either way.
5. **Reconcile the deletion promise's scope with Life Gaps/waitlist records that are not tied to an account**, so a member requesting account deletion understands whether that also covers a same-email quiz submission or waitlist entry, since today it structurally would not.

### Documented-but-unverified behavior (needs a live check, not necessarily a code change)

6. **Confirm the actual foreign-key and cascade behavior of `life_graph`, `profiles`, `reports`, `documents`, `waitlist`, and `conversations`**, the six tables that predate the tracked-migration convention and have no `CREATE TABLE` statement this audit could read. Every claim in Section 2 marked "unverifiable from migrations" depends on this.
7. **Confirm whether any process, anywhere outside this codebase (a manual runbook, a support-team script, a direct Supabase-dashboard action), currently fulfills the privacy page's deletion promise in practice**, and if so, document it, since this audit found no code-level implementation but cannot rule out a fully manual, undocumented process existing today.
8. **Confirm whether Stripe customer/subscription objects are ever deleted or anonymized as part of an account-deletion request**, which is outside this audit's read-only, no-Stripe-access scope to verify directly.

### Confirmed-safe behavior (do not need further work to be beta-ready as-is)

9. **The cancellation-to-grace-period timing is correct and matches the stated 30-day window exactly.**
10. **Access is correctly and immediately revoked when a subscription is actually deleted in Stripe**, not left open through an unintended extra grace window.
11. **Calendar OAuth tokens and imported events cascade correctly** and would be fully removed if an account deletion ever succeeds.
12. **Core financial and planning tables added since the migration convention began** (tool results, retirement plans, net worth snapshots, payday check-ins, caregiving ledgers, guide purchases and plans, Guide conversations) **all cascade correctly from `auth.users`.**
13. **Adult-migration claims correctly cascade on the parent side and correctly nullify (rather than block or leak) on the migrated-adult side.**

**No item in this checklist was acted on by this audit.** Every finding reflects read-only inspection of the code and schema as committed, not a change made during this review.
