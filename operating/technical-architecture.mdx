## Context & Philosophy

The Krovos architecture is designed around four principles:

1. **Life Graph first:** member facts and saved outcomes are reusable across the product.
2. **Solo-founder sustainable:** managed services, one application repository, and deterministic calculations where an additional model call is unnecessary.
3. **Privacy by architecture:** member data is never sold or used for advertising; documented processors receive only what is needed to perform the requested feature.
4. **Permission-gated households:** connection does not imply access. Each shared domain has an explicit control and RLS enforcement.

## Data Architecture

### The Life Graph Schema

The Life Graph is the central member-context record. `life_graph` stores flexible `financial`, `family`, `career`, `household`, `goals`, and `history` JSONB domains. Purpose-built relational tables surround it where Krovos needs queryable history, permissions, purchases, usage accounting, recurrence, or durable outcomes.

The linked production database contained the following 41 public tables when verified July 23, 2026.

### Household and Life Labor

| Table | Key data | Production purpose |
|---|---|---|
| `household_connections` | primary and connected users, invitation and status | Explicit opt-in household relationship |

| `life_labor_settings` | household names, paid hours, manual partner income | Life Labor household configuration |

`life_labor_tasks.task_source` separates `household`, `personal`, `work`, and `guide_linked` items. One-time dated tasks use `occurrence_date` on the parent row. Daily, weekly, biweekly, and monthly items use `life_labor_task_occurrences`. Household RLS follows the task source. Household tasks require `life_labor_shared` to read and `life_labor_edit_access` to write. Personal calendar items follow `calendar_shared`, including collaborative editing when enabled. Work and Guide-linked tasks remain owner-only.

### Independent-Adult Account Carryover

Once a tracked child is 18, the parent can generate a 45-day claim code in Settings. The signup flow attaches a valid code to the new account without consuming it. A claim-backed subscription receives no free trial; Stripe Checkout must report a paid session before carryover begins.

Carryover is intentionally narrow: name, date of birth, relationship provenance, and physical copies of documents attached specifically to that child. The copied files receive paths inside the new member's own private storage folder. Parent finances, household 529 records, tax-dependent data, child-related spending, siblings, and all other household information remain behind. The new member is not added to a household connection and shares nothing back automatically.

The signed Stripe webhook performs the carryover and marks the claim used only after the Life Graph and document copies succeed. Failed processing returns an error so Stripe can retry. A used claim cannot be regenerated for the same child.

### Security Model

- Row-level security (RLS) enforced at the database level, users can only access their own data
- Documents stored in user-scoped Supabase storage buckets with RLS policies
- SSNs and government ID numbers excluded from extraction prompt, permanently
- No sale or advertising use. Selected data is sent only to documented service providers necessary to operate Krovos, including Anthropic for extraction, Life View generation, and personalized Guide responses.
- Standard Anthropic commercial API retention terms apply unless Krovos obtains a separate written zero-data-retention agreement. Krovos must not claim zero retention without that agreement.
- Krovos Guide sends member-entered messages, recent bounded conversation context, privacy-minimized structured Life Graph facts, and relevant allowlisted outcome summaries. It does not automatically attach raw documents, raw tool inputs, credentials, or government identification numbers. Members are instructed not to enter those details in chat.

## AI Architecture

### Document Extraction Pipeline

The extraction pipeline reads uploaded documents and populates the Life Graph automatically. It is the primary white-glove mechanism, the product does the work of understanding so the user doesn't have to.

**1. Document Download**
API route fetches uploaded documents from Supabase Storage using the service role key. Multiple documents processed in a single API call.

**2. Base64 Encoding**
Documents converted to base64 for inclusion in the Claude API request. PDFs use `application/pdf` media type. Images use `image/jpeg` or `image/png`.

**3. Claude Extraction**
`claude-sonnet-4-6` reads all documents simultaneously with a structured extraction prompt. Returns a JSON object with all extracted fields. The prompt includes `userName` to correctly assign income to the right person.

**4. Error Handling**
If a document causes an API error, the pipeline retries with PDFs only. Individual document failures are logged and skipped without killing the extraction.

**5. Life Graph Update**
Extracted data merged into the `life_graph.financial` JSONB object. Existing user-confirmed data takes precedence over newly extracted data to preserve user corrections.

### Report Generation Pipeline

**1. Life Snapshot Assembly**
Server-side code assembles a `lifeSnapshot` object from the Life Graph, calculating ages server-side, computing annual net estimates, separating partner income, aggregating investment totals. Never lets Claude infer ages from DOBs.

**2. Prompt Construction**
Prompt includes CRITICAL RULES section that explicitly prevents common failure modes: single earner misidentification, generic advice. Includes the full `lifeSnapshot` as structured JSON.

**3. Claude Generation**
`claude-sonnet-4-6` with `max_tokens: 6000` generates a structured JSON report. The response is parsed and validated before storage.

**4. Report Storage**
Generated report stored in `reports` table as JSONB. The paginated report display reads sections individually for performance.

### Krovos Guide

Krovos Guide uses the same Anthropic commercial API but a separate context and usage path:

1. The authenticated member chooses or creates a `guide_conversations` record.
2. Krovos assembles the current message, a bounded number of recent `guide_messages`, privacy-minimized allowlisted Life Graph facts, and relevant saved outcomes.
3. Raw documents, credentials, government identification numbers, and unrestricted raw tool inputs are not automatically attached.
4. The response and token counts are saved to `guide_messages`.
5. `guide_usage_monthly` enforces daily turns, monthly turns, and monthly estimated-cost limits before another request is made.

The limits are environment-configurable. Stored conversation history increases database storage and context tokens only when messages are included in the bounded context window; the application does not resend an unlimited transcript.

### Deterministic Personalization

Guide plans, Core tool results, Retirement scenarios, Life Updates, Life Labor, Life Calendar recurrence, and most progress calculations operate from stored member facts without another Anthropic request. The model is reserved for work that requires interpretation or generation rather than ordinary calculations, persistence, or routing.

## Frontend Architecture

### Design System

The Krovos design system is implemented in Tailwind CSS with two intentional themes. Midnight Slate remains primary. Daylight Lantern is the light counterpart. The theme control exposes only dark and light; there is no user-facing auto/system third state.

| Token | Value | Tailwind Class |
|---|---|---|
| Primary background | #1B2230 (Midnight Slate) | `bg-[#1B2230]` |
| Primary accent | #F6C56A (Lantern Gold) | `text-[#F6C56A]`, `bg-[#F6C56A]` |
| Secondary accent | #E07A3F (Ember Orange) | `text-[#E07A3F]` |
| Informational | #7FA6BB (Mist Blue) | `text-[#7FA6BB]` |
| Text / surfaces | #E9EEF2 (Soft Fog) | `text-[#E9EEF2]`, `text-[#E9EEF2]/60` |

Daylight Lantern remaps the same brand system to warm light surfaces, dark readable text, Lantern Gold emphasis, Mist Blue information, and Ember Orange risk/attention states. Shared theme rules cover the application shell and legacy color utilities; bespoke SVG and route-specific compositions still require explicit theme treatment.

### Key Components

- **Root application shell**: Linked Krovos identity, Today, Goals, Calendar, Ask Krovos, Explore, theme selection, Settings, and route-aware return navigation.
- **KrovosIcon.tsx**: Reusable lantern SVG component.
- **KrovosLoader.tsx**: Animated flame loading screen. Fixed position overlay. Used across all pages for loading states.
- **useLifeGraph hook**: Custom React hook wrapping Supabase queries. Handles load, save, and error state for the Life Graph. Restores saved form state on back navigation.
- **GuideGate**: Enforces Guide ownership or allowed household inheritance before Guide-exclusive tools render.
- **FeedbackButton**: Root-level authenticated feedback entry with optional rating, category, text, and private screenshots.

### Accessibility Standard

The shared shell supplies focus-visible states, route-aware navigation, recoverable loading/error/not-found states, and reduced-motion handling. Automated audits cover app-shell reachability, tool input behavior, representative Core responsive states, and Retirement boundaries. A final route-by-route human accessibility and novice-comprehension pass remains a pre-beta release gate; this document does not claim complete WCAG certification.

### Migration Operations

The Supabase CLI is linked. Timestamped migrations from the July 22 series forward are recorded in production history, including the recurring Life Calendar occurrence schema and the Retirement grant repair. A targeted SQL file can be executed against the linked project and then repaired into migration history when the legacy backlog blocks a normal push.

Older pre-July-22 migrations include short and duplicate date versions that were historically applied through the SQL Editor without matching CLI history. Therefore `supabase db push --include-all` remains unsafe until that legacy history is reconciled file by file. This is a legacy-history limitation, not an absence of CLI or production database access.
